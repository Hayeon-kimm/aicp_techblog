## 요약

PPTX 파일의 내부 XML 구조를 직접 탐색하고, 상용 AI 툴(GenSpark)의 코드를 분석하여 웹 기반 슬라이드 생성 도구의 렌더링 매커니즘을 파악했어요. '스마트 플랜트 전력 안정화'를 주제로 컨설팅 스타일의 슬라이드 초안을 생성하고 프롬프트 엔지니어링을 수행했지만, XML 직접 수정 시 좌표계(EMU)와 태그 구조의 복잡성으로 인한 한계를 확인했어요.

## 환경

- OS: Windows
- 분석 대상: .pptx 파일 (ZIP 기반 XML 구조)
- 활용 도구: GenSpark, Gemini

## 오늘 한 일

### 1. PPTX 파일 구조 리버스 엔지니어링
- `.pptx` 파일의 확장자를 `.zip`으로 변환하여 내부 XML 구조를 직접 탐색했어요.
- 슬라이드의 디자인과 데이터가 저장되는 `ppt/slides/` 경로의 매커니즘을 파악했어요.

### 2. 상용 AI 툴(GenSpark) 코드 분석
- 젠스파크에서 생성된 슬라이드 소스 코드를 분석했어요.
- 웹 기반 툴들이 디자인을 렌더링하기 위해 HTML/CSS를 중간 언어로 활용한다는 인사이트를 얻었어요.

### 3. 디자인 거버넌스 수립 및 실습
- '스마트 플랜트 전력 안정화'를 주제로 컨설팅 펌 스타일의 슬라이드 초안을 생성했어요.
- AI가 생성한 결과물의 시각적 퀄리티를 제어하기 위한 프롬프트 엔지니어링을 수행했어요.

## 에러/이슈

### 1. XML 직접 수정의 높은 진입장벽
- 실제 PPT XML은 좌표계(EMU)와 태그 구조가 매우 복잡해서 수동으로 수정할 경우 구문 오류(Syntax Error)가 발생하기 쉬운 구조임을 확인했어요.

### 2. XML 핸들링 미숙
- 제미나이 등을 활용해 XML 파일을 직접 수정하여 다시 압축(re-zip)하려 했으나, 텍스트 데이터 외에 레이아웃 좌표값이나 태그 닫힘 문제 등으로 인해 파일이 정상적으로 열리지 않는 이슈가 발생했어요.

### 3. 확장자 가시성 문제
- 윈도우 환경에서 파일 확장명이 숨겨져 있어 초기 `.zip` 변환 과정에서 혼동이 있었으나, 설정을 통해 해결했어요.

## 해결 방법

- 직접적인 코드 수정 대신 젠스파크의 내부 편집 기능을 활용해 디자인을 완성한 뒤, 완성된 결과물의 XML 구조를 역으로 대조하며 태그의 위치를 파악하는 방식으로 접근했어요.
- 즉, **"결과물 → XML 역추적"** 방식이 **"XML 직접 수정 → 결과물"** 방식보다 훨씬 효율적이었어요.

## 참고

- PPTX 파일 내부 구조: `ppt/slides/` 디렉토리에 각 슬라이드의 XML이 저장됨
- EMU(English Metric Units): PowerPoint XML에서 사용하는 좌표 단위 (1인치 = 914400 EMU)
- GenSpark: 웹 기반 AI 슬라이드 생성 도구로, HTML/CSS를 중간 렌더링 언어로 활용
